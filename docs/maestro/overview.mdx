##  Descrição do Produto

**Maestro** é uma plataforma inteligente desenvolvida como um orquestrador de campanhas, capaz de **gerenciar templates** e **coordenar o disparo de mensagens multicanal**, integrando-se diretamente com outros brokers de envio.  

Seu propósito é **centralizar, automatizar e qualificar a comunicação em escala**, otimizando o processo de envio de mensagens.

---

##  Principais Funcionalidades

- Centralização e unificação de templates em um único ambiente  
- Gestão do fluxo de aprovação, com suporte a múltiplos perfis de usuários  
- Aplicação de regras de envio personalizadas, via **Motor de Regras**  

---

##  Canais Compatíveis

- WhatsApp  
- E-mail  
- SMS  

---

##  Como Funciona

O Maestro opera como um **hub** entre:  
- As assessorias responsáveis pelos envios  
- Os motores de validação  
- Os brokers de entrega  

A plataforma foi projetada para ser **extensível**, permitindo futura integração com:  
- Novos validadores  
- Brokers adicionais  
- Mecanismos de análise  

---

##  Perfis de Uso

O Maestro contará com interfaces específicas para diferentes perfis:  

- **Gerentes do ambiente** - Empresa administradora e supervisora do ambiente
- **Assessorias**  
- **Revisora** - Empresa responsável pela revisão e aprovação de templates

##  Tela de Login

A tela de login do Maestro é responsável pelo **gerenciamento de acesso e autenticação de usuários**, oferecendo dois mecanismos complementares:
- Login por e-mail e senha
- Login com SSO (Single Sign-On)

---

### SSO (Single Sign-On)

A autenticação pode ser configurada para permitir **login via credenciais corporativas**.  

- Permite integração com **Azure Active Directory (Azure AD)**.  
- É necessário inserir o **ID da aplicação** no Azure AD, que identifica o sistema dentro do diretório.  
- Usuários podem acessar o sistema **sem criar novas senhas**, utilizando suas credenciais da organização.  

Essa configuração proporciona **maior segurança e praticidade**.

Para que o SSO seja efetivado, deve-se liberar a URL `https://api.maestro.robbu.global/v1/auth/login/sso/callback` em seu aplicativo de SSO para que o retorno possa ser validado

O login SSO valida usuários cadastrados dentro da plataforma Maestro, verifique se o usuário que irá realizar o login por SSO já está cadastrado na tela de usuários.

##  Público

A **importação de público** é um recurso que será utilizado como ponto de partida para campanhas massificada

Esse recurso garante maior agilidade na preparação de campanhas e na gestão contínua da base de público.

Dessa forma, um mesmo público pode ser utilizado por mais de uma campanha, se necessário.

![importar publico](https://robbublob.blob.core.windows.net/docs/maestro/importpublico.png)

O layout padrão do arquivo mailing possuí as seguintes colunas: 

```
TIPO_REGISTRO, VALOR_REGISTRO, NOME_CLIENTE, PRIORIDADE, CPF_CNPJ, COD_CONTRATO
```

Para utilizar variáveis dinâmica crie-as após a última coluna do arquivo.
Tudo que estiver após a coluna COD_CONTRATO são campos adicionado manualmente pelo usuário como no exemplo abaixo.

```
TIPO_REGISTRO, VALOR_REGISTRO, NOME_CLIENTE, PRIORIDADE, CPF_CNPJ, COD_CONTRATO, VAR1, VAR2, VAR3
```

###  Conta 

#### Configurações de Templates

A configuração de templates permite definir regras relacionadas ao uso e validade dos modelos disponíveis no sistema. 
É possível estabelecer um tempo de expiração para os templates, garantindo que conteúdos desatualizados não sejam utilizados. 
Para isso, defina o período de validade (em dias), após o qual o template será considerado expirado. 
Templates expirados ainda poderão ser visualizados, mas não estarão disponíveis para novas utilizações até que sejam atualizados ou revalidados.
![config conta](https://robbublob.blob.core.windows.net/docs/maestro/configconta.png)

> ⚠️ **Atenção:** o sincronismo de expiração de todos os templates ocorre **uma vez ao dia, durante o período noturno**.

### Integração com provedores
Esta seção permite que cada assessoria integre os seus provedores.
Nesta seção também, a Gerente de Ambiente pode visualizar os provedores que cada assessoria usa, sem a possibilidade de visualizar as credenciais.

####  Como Funciona?

1. O provedor fornece um **token de autenticação** (pode ser um **JWT**, **Bearer Token** ou uma **chave fixa**).  
2. Esse **token** deve ser incluído na configuração deste provedor
3. O sistema envia a mensagem para o **endpoint da API do provedor**.  
4. O provedor processa e entrega a mensagem ao destinatário.  
5. Uma resposta é retornada, indicando o **status do envio** (sucesso, falha, pendente, etc.).
