  ## Encerramento automático de atendimentos por inatividade

  Para oferecer mais controle sobre a gestão de atendimentos e evitar o acúmulo de protocolos sem retorno, o Invenio passa a permitir a **configuração do encerramento automático de contatos inativos**. Com essa melhoria, cada operação pode definir por quanto tempo um atendimento pode permanecer sem interação antes de ser encerrado automaticamente.

  ---

  ## O que mudou

  Anteriormente, o sistema utilizava uma regra fixa para encerramento de protocolos inativos. Agora, é possível configurar esse prazo de forma flexível em dois níveis:

  - **Conta** (regra geral)
  - **Carteira (Segmento)** (regra específica)

  Quando existir uma configuração na carteira, ela terá prioridade sobre a configuração da conta.

  ---

  ## Como funciona

  O sistema monitora automaticamente o tempo desde a **última mensagem enviada na conversa**, independentemente de quem enviou. Se o prazo configurado for atingido:

  - O protocolo é encerrado automaticamente
  - O contato é removido da fidelização (quando aplicável)
  - Pendências são finalizadas
  - Um evento de encerramento é registrado no histórico

  Essa rotina é executada automaticamente pelo sistema.

  ---

  ## Configuração na Conta

  O administrador pode definir um prazo padrão de inatividade que será aplicado a todos os segmentos que não possuam configuração própria.

  **Local:** Configurações da Conta

  **Campo:** Prazo de encerramento por inatividade (em dias)

  **Regras:**

  - Valor permitido: **1 a 30 dias**
  - Campo obrigatório

  ---

  ## Configuração na Carteira (Segmento)

  Cada carteira pode usar a configuração padrão da conta ou definir um prazo específico.

  ### Opção: Usar configurações da conta

  Quando ativada:

  - O segmento utiliza automaticamente o prazo definido na conta.

  Quando desativada:

  - O campo de dias fica disponível para configuração própria.

  **Valor permitido:**\
  1 a 30 dias

  ---

  ## Regra de prioridade

  A aplicação das configurações segue a seguinte ordem:

  1. Configuração da **Carteira (Segmento)**
  2. Configuração da **Conta**

  Se o segmento não possuir valor definido, o sistema utiliza automaticamente o valor da conta.

  ---

  ## Benefícios

  - Reduz acúmulo de atendimentos inativos
  - Mantém a base de dados mais organizada
  - Diminui esforço operacional manual
  - Permite ajustes conforme a realidade de cada operação

  ---

  ## Observações importantes

  - Apenas protocolos sem interação dentro do período configurado serão encerrados.
  - Protocolos dentro do prazo permanecem ativos normalmente.
  - A configuração não impacta atendimentos já encerrados.
</Update>

  ## Press Release —Indicador de Digitação no WhatsApp (Invenio Live)

  ### Visão Geral

  O Indicador de Digitação foi implementado no Invenio Live para tornar o atendimento via WhatsApp mais fluido e natural, aproximando a experiência do cliente final ao comportamento nativo do aplicativo.

  Agora, quando o atendente estiver digitando uma resposta, o cliente visualizará no WhatsApp a indicação de que sua mensagem está sendo respondida em tempo real.

  ### Essa melhoria aumenta:

  - Sensação de agilidade
  - Transparência na comunicação
  - Redução da percepção de demora
  - Experiência mais humanizada

  ### Objetivo da funcionalidade

  Evitar que o cliente interprete pausas na resposta como ausência de atendimento, exibindo a indicação de que o atendente está digitando.

  ### Como funciona

  - Início da digitação

  #### Quando:

  - O atendente está em uma conversa ativa no Invenio Live
  - E começa a digitar uma mensagem

  #### O sistema:

  - Envia automaticamente uma notificação ao WhatsApp
  - O cliente visualiza o indicador de “digitando...”
  - Pausa na digitação

  #### Quando:

  - O atendente para de digitar

  #### O sistema:

  - Interrompe o envio da notificação
  - O indicador desaparece automaticamente no WhatsApp

  A notificação deixará de aparecer em no máximo 25 segundos, caso nenhuma nova digitação seja detectada.

  ### Onde se aplica

  Esta funcionalidade está disponível para:

  - Conversas via WhatsApp
  - Utilizando o provedor Positus
  - Realizadas pelo Invenio Live

  #### Outros canais não são impactados.

  ### Regras de funcionamento

  O envio do indicador respeita um intervalo mínimo entre chamadas, evitando requisições excessivas. Chamadas repetidas não são disparadas enquanto a digitação continua. Caso ocorra erro técnico na comunicação com o provedor, o atendimento continua normalmente (sem impacto no envio de mensagens).

  ### Experiência para o cliente final

  #### O cliente perceberá:

  - Que o atendente está ativo
  - Que sua mensagem está sendo respondida
  - Uma experiência mais próxima do uso direto do WhatsApp
  - Nenhuma ação é necessária por parte do cliente.

  ### Impacto para o usuário

  Atendentes não terão acesso a templates de autenticação no Live Não há impacto no envio de mensagens comuns ou outros tipos de templates Fluxos automatizados continuam funcionando normalmente

  ## Press Release — Melhoria no Agrupamento de Tempo de Espera para Agrupar Mensagens de Entrada

  ### Mais fluidez e inteligência no tratamento de mensagens sequenciais

  O Invenio recebeu uma melhoria importante no Webchat para tornar as conversas mais naturais e eficientes. Agora, é possível configurar um **tempo de espera para agrupar mensagens enviadas em sequência**, evitando que mensagens fragmentadas sejam tratadas como interações separadas pelo chatbot.

  Essa evolução atende a cenários comuns em canais digitais, nos quais os usuários enviam mensagens em partes curtas, como:

  **Exemplo:**\
  “Oi”\
  “Bom dia”

  Com a nova configuração, o sistema pode agrupar automaticamente essas mensagens em uma única interação:

  **Resultado:**\
  “Oi Bom dia”

  ---

  ## Como funciona

  Foi adicionada uma nova configuração que permite definir, em segundos, quanto tempo o chatbot deve aguardar antes de processar as mensagens recebidas.

  - O intervalo pode ser configurado entre **0 e 5 segundos**.
  - Mensagens enviadas dentro desse período são automaticamente combinadas.
  - A configuração é independente para **canais online** e **canais offline**.
  - Caso o campo fique vazio, o comportamento padrão será **0 segundos** (sem agrupamento).

  Essa abordagem reduz respostas fragmentadas e melhora a interpretação do contexto pelo bot.

  ---

  ### Benefícios para a operação

  - Conversas mais naturais e organizadas
  - Redução de interpretações incorretas pelo chatbot
  - Melhor aproveitamento dos fluxos automatizados
  - Experiência mais fluida para o usuário final

  ---

  ### Experiência de configuração mais clara

  Além da nova funcionalidade, o campo de configuração foi atualizado para tornar o uso mais intuitivo:

  - Limite máximo definido em **5 segundos**
  - Ajuste automático caso seja informado valor acima do permitido
  - Nova legenda orientativa para facilitar o entendimento

  ---
</Update>

  ## Bloqueio de recebimento de arquivos por extensão

  A funcionalidade **Bloquear recebimento de arquivos por extensão** permite impedir automaticamente o recebimento de arquivos com extensões consideradas indesejadas ou inseguras.

  O bloqueio:

  - Se aplica a **todos os canais**
  - Mantém a mensagem registrada na conversa
  - Impede o download do conteúdo
  - Pode ser configurado no nível da **Conta** ou do **Segmento**
  - Respeita hierarquia de prioridade (**Segmento \> Conta**)

  ---

  ### Como funciona a hierarquia

  A regra aplicada no recebimento do arquivo segue esta ordem:

  1. Se o **segmento possuir configuração própria**, ela será aplicada.
  2. Caso contrário, será utilizada a **configuração definida na conta**.

  > A configuração do segmento sempre sobrescreve a da conta quando existir.

  ---

  ## Configuração no nível da Conta

  Define uma política padrão para todos os segmentos que não possuam regra específica.

  ### Caminho

  `Configurações da Conta → Bloqueio de arquivos de entrada`

  ### Como configurar

  - Selecione as extensões que deseja bloquear.
  - Utilize o campo de busca para localizar extensões específicas.
  - Utilize:
    - **Marcar todos**
    - **Desmarcar todos**

  As extensões selecionadas passam a ser a **regra padrão da conta**.

  ---

  ## Configuração no nível do Segmento

  Permite sobrescrever a política da conta quando necessário.

  ### Caminho

  `Segmentos → Selecionar segmento → Bloqueio de arquivos de entrada`

  ## Toggle: Usar configurações gerais da conta

  ### Toggle ativado

  - O segmento utiliza automaticamente as regras definidas na conta.
  - A lista de extensões não é exibida.

  ### Toggle desativado

  - A lista de extensões é exibida.
  - O administrador pode definir regras próprias para o segmento.

  > Se o toggle estiver desativado e nenhuma extensão estiver selecionada, o segmento permitirá o recebimento de qualquer tipo de arquivo.

  > A configuração da conta só será considerada quando o toggle estiver ativado.

  ---

  ### Comportamento no recebimento do arquivo

  Quando um contato envia um arquivo:

  1. O sistema identifica o segmento da conversa.
  2. Aplica a regra do segmento (se existir).
  3. Caso não exista regra no segmento, aplica a regra da conta.
  4. Se a extensão estiver bloqueada:
     - O arquivo não é armazenado.
     - O download não é permitido.
     - A mensagem permanece registrada na conversa.

  ---

  ## Análise Antivírus em Tempo Real para Mensagens Recebidas.

  Verificação de Vírus em Arquivos Este recurso foi criado para aumentar a segurança no recebimento e download de arquivos, protegendo os usuários contra possíveis ameaças.

  1. Recebimento e Verificação de Arquivos Sempre que um arquivo é recebido:

  O sistema realiza automaticamente uma verificação de segurança O arquivo é classificado como:

  - Seguro → segue normalmente
  - Suspeito → recebe sinalização de risco Arquivos suspeitos continuam disponíveis, porém com tratamento diferenciado de segurança.

  2. Identificação de Arquivos Suspeitos\*\*

  Quando um arquivo suspeito é exibido:

  - Ele aparece com um **aviso visual de risco**
  - Esse comportamento é o mesmo em:
    - Invenio Center
    - Invenio Live\
      📌 Importante:\
      A verificação se aplica apenas a arquivos recebidos **após a ativação da funcionalidade**.

  3. Controle de Permissão do Usuário A liberação para download de arquivos suspeitos depende das permissões do usuário. Permissão: “Permitir by-pass para download de arquivos suspeitos”

  - Desativada (padrão): O usuário não pode baixar arquivos suspeitos O download permanece bloqueado
  - Ativada: O usuário pode optar por baixar o arquivo, mediante confirmação de risco

  4. Bloqueio de Download por Segurança Por padrão, o sistema protege o usuário:

  Downloads de arquivos suspeitos são bloqueados automaticamente 📌 Mensagem exibida: "O download deste arquivo foi bloqueado por motivos de segurança."

  5. Download com Confirmação de Risco Se o usuário tiver permissão para prosseguir:

  Será exibida uma tela de confirmação antes do download

  - Mensagem apresentada: "Este arquivo foi identificado como suspeito e pode representar riscos à segurança. Ao prosseguir, você assume total responsabilidade pelo download."
  - Opções disponíveis: Cancelar → interrompe a ação Baixar mesmo assim → prossegue com o download ⚠️ Essa ação exige confirmação explícita e indica que o usuário está ciente do risco.

  6. Registro de Segurança (Auditoria) Sempre que um arquivo suspeito for baixado:

  O sistema registra informações da ação, como:

  - Usuário que realizou o download
  - Identificação do arquivo
  - Referência do conteúdo baixado

  ### Esse registro:

  - Não gera notificações
  - Não interfere no fluxo É utilizado apenas para controle e auditoria de segurança

  ## Cálculo e Compartilhamento de Limites de Mensagens por Portfólio

  A partir de hoje, 7 de outubro de 2025, os limites de mensagens são calculados e definidos com base no portfólio de negócios vinculado à BM e serão compartilhados por todos os números de telefone comerciais.

  Os portfólios de negócios existentes terão seu limite de mensagens definido para o limite mais alto de qualquer número de telefone em seu portfólio (por exemplo, se um portfólio tiver um número de telefone com um limite de 100 mil e outros números de telefone com limites mais baixos, o limite do portfólio será definido como 100 mil).

  Essas mudanças resultarão em:

  - Acesso mais rápido a limites mais altos, com upgrades em 6 horas (em comparação com as 24 horas atuais) e acesso imediato aos limites da carteira para números recém-registrados.
  - Maior flexibilidade, permitindo que você tenha quantos telefones forem necessários sem a necessidade de telefones adicionais para acessar limites mais altos, já que todos os telefones compartilham o limite do portfólio.
  - Redução da carga administrativa e dos custos operacionais, minimizando a necessidade de números de telefone desnecessários.

  Você pode acompanhar o limite atual do portifólio pelo Center acessando: Canais \> WhatsApp \> Waba \> Números \> Coluna Limite de mensagens.

  - Observação: O valor exibido não é o limite diário do número, mas sim de todos os números do mesmo portifólio.

  Fonte: [https://developers.facebook.com/docs/whatsapp/messaging-limits](https://developers.facebook.com/docs/whatsapp/messaging-limits) [https://developers.facebook.com/docs/whatsapp/messaging-limits/upcoming-changes](https://developers.facebook.com/docs/whatsapp/messaging-limits/upcoming-changes)
</Update>

  ## Tratamento de debounce no Invenio Center:

  Center \> Menu do Usuário\> Configurações \> Conta \> Tempo de espera (segundos) para tratar mensagens entrantes O Invenio Center passa a suportar o tratamento de debounce condicional no motor de atendimento IDR.

  ### O que é debounce?

  É um recurso que define um tempo de espera (em segundos) para que várias mensagens enviadas em sequência sejam entendidas como uma única entrada. Isso evita duplicidade e garante mais fluidez no atendimento. Como funciona agora Linhas atualizadas (nova versão da IDR)

  O tempo de espera é aplicado conforme a configuração da linha. Exemplo: se o tempo estiver configurado em 3 segundos, duas mensagens enviadas dentro desse intervalo serão agrupadas e tratadas como uma só.

  Linhas antigas (versão anterior da IDR) O recurso não é aplicado. Todas as mensagens continuam sendo processadas individualmente, preservando o comportamento atual.

  Diferença por tipo de canal Canais online: 3 segundos Canais offline: 10 segundos Benefícios Atendimento mais estável e organizado Ajuste flexível por canal Compatibilidade com linhas que ainda utilizam o modelo antigo Gostaria de ativar esta funcionalidade agora?

  Entre em contato com o seu gerente de contas para solicitar a habilitação. Caso já utilize a IDR na versão V1, será necessário migrar para a versão V2 como parte do processo de ativação.

  ## Palavras Reservadas

  Os endpoints possuem algumas constantes e funções predefinidas que são fundamentais para seu funcionamento. Portanto, **não é possível criar novas constantes** com as seguintes palavras reservadas:

  - `logger`
  - `loggerHistory`
  - `response`
  - `request`

  Caso você tente definir uma constante com alguma dessas palavras reservadas, o seu endpoint **retornará erro**.

  ## Limitações

  Os endpoints possuem as seguintes limitações:

  - **Tempo máximo de execução:** 10 segundos. Se ultrapassar esse limite, será exibido o erro:
    ```json
    {
      "error": "The JavaScript code took more than 10 seconds to execute"
    }
    ```
  - **Obrigatoriedade de retorno:** É necessário sempre retornar algo no endpoint utilizando o método `response()`. Caso contrário, será exibido o erro:
    ```json
    {
      "error": "Endpoint has no data returned"
    }
    ```

  ## Variáveis Globais

  Por padrão, existem algumas variáveis essenciais para o funcionamento dos endpoints:

  ### `request`

  Variável que contém várias informações sobre a requisição, como:

  - `method` → Método da requisição (GET, POST, etc.).
  - `headers` → Cabeçalhos da requisição.
  - `body` → Corpo da requisição.
  - `query` → Parâmetros da query string.

  #### Exemplos de Uso:

  ```javascript
  if (request.method === "get") {}
  
  if (request.headers.hasOwnProperty("server")) {}
  
  if (request.body.user.status === "active") {}
  
  if (request.query.user.q === "Nome") {}
  ```

  ## Métodos

  Existem alguns métodos fundamentais para o funcionamento de um endpoint:

  ### `logger(data: any)`

  Método utilizado para adicionar dados ao **logger** do request, útil no processo de debugging.

  #### Exemplos:

  ```javascript
  logger(request)
  logger(request.body.user)
  logger("Testando o log")
  logger({data: request, tipo: "Chamada externa"})
  ```

  ### `response(data: object, status: number = 200, success: boolean = true)`

  Método utilizado para **retornar a resposta do endpoint**.

  #### Exemplos:

  ```javascript
  // Retorna sucesso
  return response({success: true})
  
  // Retorna erro
  return response({error: true}, 400)
  ```
